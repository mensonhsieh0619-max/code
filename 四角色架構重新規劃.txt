AI Health Platform — 四角色架構重新規劃 Prompt
產生日期：2026-07-10
用途：作為重新實作四個角色（使用者／教練／營養師／管理員）導覽與頁面結構的依據。

────────────────────────────────────────────────────────────────
【核心架構原則】（四個角色的規劃都必須遵守）
────────────────────────────────────────────────────────────────

1. 資料只沿著「同意／授權」流動
   使用者在「授權分享」建立授權（選對象角色 + 勾選資料範圍 + 設定期限），
   教練／營養師只能看到「被明確授權」的學員／個案，且只能看到「被授權的欄位」。
   這套機制目前實作正確（getAuthorizationsByRole()、authAllows(auth, "步數") 等
   逐欄位檢查），是整個平台的核心賣點，重新規劃時必須保留、不能破壞。

2. 一份內容只能有一個入口
   同一組資料／同一個渲染結果，不可以用不同名字包裝成兩三個導覽項目。
   （這是這次重新規劃要修正的主要問題，詳見各角色章節。）

3. 管理員 = 系統／技術層，不涉入個別病患的健康建議
   管理員看到的應該是「帳號、FHIR 資源、授權紀錄、區塊鏈紀錄、系統設定、競賽報名審核」，
   不應該出現任何「個人健康建議 / AI 風險預測 / 健康報告下載」這類以單一病患為中心的功能。

4. 每個角色的導覽項目數 = 實際不同內容的數量
   點進去內容相同的項目要合併，內容太單薄（例如只有 3 個數字）不足以撐一個獨立頁面的
   要併入其他頁面，不要為了「看起來功能很多」硬拆成多個入口。

5. 角色專屬功能只能出現在該角色的頁面
   任何新增功能都必須在建立導覽按鈕／頁面區塊之前，先判斷 getCurrentRole()，
   只允許明確需要的角色看到。不要用「所有角色都給」當預設值。

────────────────────────────────────────────────────────────────
【已知的具體證據】（重新規劃前先確認，避免砍錯或誤判）
────────────────────────────────────────────────────────────────

A. 教練「AI 訓練建議」page 與「學員運動紀錄」page 內容重複
   - shared.js renderCoachViews()：
     training-advice-panel 塞入 `analysis.exerciseAdvice`（純建議文字）
     student-exercise-panel 塞入「步數／運動時間／消耗熱量／心率／達成率＋同一段 exerciseAdvice」
   - 結論：「學員運動紀錄」內容已經包含「AI 訓練建議」的全部資訊，後者是多餘的獨立頁。

B. 營養師「BMI分析」「血壓提醒」「AI飲食建議」三個導覽項目渲染出完全相同的 HTML
   - shared.js renderCaseNutritionData()：
     產出同一個 html 變數，同時塞進：
       setHTML("bmi-analysis-panel", html)
       setHTML("diet-advice-panel", html)
       setHTML("blood-pressure-panel", html)
   - 結論：三個入口 = 一份內容，只需保留一個。

C. 管理員「帳號列表」與「使用者資料」是同一份 state.accounts，只是欄位切一半
   - 帳號列表欄位：姓名／帳號／Email／角色／狀態／操作（含啟用停用按鈕）
   - 使用者資料欄位：姓名／Email／角色／單位／手機
   - 結論：應合併成一張表：姓名／帳號／Email／角色／單位／手機／狀態／操作

D. 管理員「Observation 紀錄」內容只有 3 個 KPI 數字，資訊量撐不起獨立頁面
   - setHTML("observation-record-panel", `${kpi("健康紀錄",...)}${kpi("Observation",...)}${kpi("使用者",...)}`)
   - 結論：併入首頁 KPI 或 FHIR 紀錄頁的附加資訊，不需要獨立導覽項目。

E. 使用者「FHIR JSON」頁面（fhir-viewer-section）已經拿掉導覽入口，但頁面與程式碼還留著
   - 屬於孤兒內容，這次要嘛徹底刪除，要嘛決定保留但重新想清楚定位（見使用者章節備註）。

F. AI 對話助理／AI 交互檢查／AI 風險預測，全部是規則式模擬（非真實 AI 模型）
   - AI 助理是關鍵字比對回覆、AI 交互檢查是查表 + setTimeout 假延遲、
     AI 風險預測是線性內插算出的數字。
   - 這次規劃不要求拆掉這些功能（demo 展示合理），但四個角色頁面上都應該用一句話
     誠實標註「AI 分析為模擬展示」，避免評審誤以為串接真實模型。

────────────────────────────────────────────────────────────────
【角色一：一般使用者 User】
────────────────────────────────────────────────────────────────

角色定位：
- 平台的資料擁有者（病患本人）。記錄並查看自己的健康數據，取得 AI 個人化建議，
  決定要把哪些資料分享給教練／營養師／管理員。

核心任務：
1. 新增與查看自己的健康紀錄（血壓、體重、心率、步數、運動、睡眠等）
2. 查看 AI 針對自己健康數據的分析、建議與風險預測
3. 建立／管理授權分享（決定誰可以看哪些資料、多久）
4. 匯入穿戴裝置模擬資料
5. 管理自己的帳號資料

保留的導覽結構：
- 首頁（Hero 歡迎橫幅 + KPI 卡片 + 血壓/步數圖表 + AI摘要卡 + 通知摘要卡）
- 健康資料 ▾
  - 新增資料
  - 健康趨勢（7/14/30天/全部 圖表切換）
- AI 健康建議（飲食/運動/睡眠/就醫/生活習慣建議 + AI健康分數 + 風險標籤 + 時間軸）
- 授權分享（建立授權表單 + QR Code 顯示）
- 個人資料設定（動態注入，僅 user 角色）
- AI 風險預測（動態注入，僅 user 角色，30天趨勢預測）
- 健康報告下載（動態注入，僅 user 角色，PDF列印/下載）
- 穿戴裝置匯入（右下角浮動按鈕，不佔導覽列空間）
- 通知（右上角圖示，不用文字按鈕）

要刪除：
- FHIR JSON 檢視器（fhir-viewer-section）及其對應程式碼、渲染函式
  理由：導覽入口已移除，內容對一般使用者也難以理解，屬孤兒功能，直接刪除比留著半吊子乾淨。

待你決定的問題：
- 「競賽報名」系統（registration-section / my-registration-section）目前掛在使用者頁面，
  但性質是「參賽報名」不是「健康追蹤」，跟其他功能完全不同軸線。建議選項：
  (a) 移到獨立的報名入口頁面（跟登入頁同層級，不算在健康平台角色導覽內）
  (b) 保留在使用者頁面，但移到導覽列最後、視覺上獨立分隔
  (c) 維持現狀不動
  → 這次規劃先不動它，等你選定方向再處理。

【角色二：健身教練 Coach】
────────────────────────────────────────────────────────────────

角色定位：
- 被使用者授權查看「運動相關資料」的專業人員。管理自己的學員名單，
  依授權範圍查看學員運動數據，提供訓練建議與紀錄。

核心任務：
1. 查看被授權的學員清單與授權狀態
2. 依授權範圍查看學員的運動數據（步數/運動時間/心率/消耗熱量/達成率）並取得 AI 建議
3. 建立與查看訓練紀錄

保留的導覽結構（4 項，原本 6 項）：
- 首頁（Hero 橫幅 + KPI 卡片 + 學員運動達成率圖 + 趨勢摘要 + AI建議摘要卡）
- 學員管理 ▾
  - 學員列表（授權狀態、資料範圍、有效期限一覽表）
  - 學員總覽 ← 合併後的新頁面，取代原本「學員運動紀錄」+「學員心率趨勢」+「AI訓練建議」
    每位學員一張卡片，卡片內含：步數／運動時間／消耗熱量／心率／達成率
    （依授權欄位顯示，未授權欄位不顯示）＋ 心率趨勢小圖表 ＋ AI 訓練建議文字，
    全部整合在同一張卡片裡，不用再切換頁面
- 訓練紀錄（新增訓練紀錄表單 + 紀錄列表）
- 通知（右上角圖示）

要刪除：
- 「AI 訓練建議」獨立頁（training-advice-section）
  理由：內容與學員總覽卡片內的建議文字完全相同，屬重複頁面。
- 「學員心率趨勢」獨立頁（student-heart-rate-section）
  理由：併入學員總覽卡片內的小圖表，不需要單獨切換頁面查看。

實作提醒：
- 合併後的「學員總覽」渲染邏輯，等同把 renderStudentExerciseData() 的卡片內容
  加上一個心率小圖表 canvas，一個學員一張卡、一個 canvas id。
- renderCoachViews() 裡對 training-advice-panel 的 setHTML 呼叫要移除。

【角色三：營養師 Nutritionist】
────────────────────────────────────────────────────────────────

角色定位：
- 被使用者授權查看「營養/代謝相關資料」的專業人員。管理自己的個案名單，
  依授權範圍查看個案的體重、BMI、血壓數據，提供飲食建議。

核心任務：
1. 查看被授權的個案清單與授權狀態
2. 依授權範圍查看個案的營養健康數據（體重/BMI/血壓/營養風險）並取得 AI 飲食建議
3. 查看個案體重長期趨勢圖
4. 建立與查看營養紀錄

保留的導覽結構（4 項，原本 6 項）：
- 首頁（Hero 橫幅 + KPI 卡片 + BMI分佈圖 + 血壓風險分佈圖 + 風險摘要 + AI建議摘要卡）
- 個案管理 ▾
  - 個案列表（授權狀態、資料範圍、有效期限一覽表）
  - 個案總覽 ← 合併後的新頁面，取代原本「BMI分析」+「血壓提醒」+「AI飲食建議」
    （三者原本就是同一份渲染結果，直接合併，不需要額外開發，只是砍掉兩個多餘入口）
  - 體重趨勢（個案體重長期趨勢圖，內容獨立，保留）
- 營養紀錄（新增營養紀錄表單 + 紀錄列表）
- 通知（右上角圖示）

要刪除：
- 「BMI分析」「血壓提醒」「AI飲食建議」三個導覽項目與對應 section，
  只留其中一個（建議命名為「個案總覽」），renderCaseNutritionData() 只需
  setHTML 到保留下來的那個 panel id 即可，其餘兩個 setHTML 呼叫刪除。

【角色四：系統管理員 Admin】
────────────────────────────────────────────────────────────────

角色定位：
- 平台的系統維運者，管理帳號、監控 FHIR／區塊鏈技術資源、審核競賽報名、
  維護系統設定。不涉入任何個別病患的健康建議內容。

核心任務：
1. 管理所有帳號（建立/檢視/啟用停用），檢視完整使用者資料
2. 監控 FHIR 資源、Observation 統計、授權紀錄、區塊鏈紀錄
3. 驗證 FHIR Bundle 格式（FHIR 驗證工具）
4. 審核競賽報名
5. 維護系統設定（FHIR連線設定、AI敏感度門檻、維護模式、重置Demo資料）

保留的導覽結構（7 項，原本 9 項）：
- 首頁（Hero 橫幅 + KPI 卡片 + 帳號角色分佈圖 + 報名審核狀態圖 + 快速前往卡片 + 系統說明）
- 帳號管理 ← 合併後的新頁面，取代原本「帳號列表」+「使用者資料」
  單一表格欄位：姓名／帳號／Email／角色／單位／手機／狀態／操作（啟用停用）
- FHIR ▾
  - FHIR 紀錄
  - 授權紀錄
  - 區塊鏈紀錄
  （「Observation 紀錄」併入 FHIR 紀錄頁面上方，變成該頁的附加 KPI 小卡，不再是獨立項目）
- FHIR 驗證（動態注入，僅 admin 角色）
- 系統設定
- 快速前往卡片內另外保留：報名列表、報名審核（已是既有安排，維持）
- 通知（右上角圖示）

要刪除：
- 「使用者資料」獨立頁（user-management-section），資料併入合併後的「帳號管理」表格
- 「Observation 紀錄」獨立頁（observation-record-section），
  3 個 KPI 數字改成附加在「FHIR 紀錄」頁面最上方即可

實作提醒：
- account-management-body 的 setHTML 內容要擴充欄位（加上單位、手機），
  user-management-body 這個 id 與其對應的 setHTML 呼叫可以整個刪除。
- observation-record-panel 的三個 kpi() 呼叫，改成插入 fhir-record-section 內，
  observation-record-section 這個 section 與其導覽項目一起刪除。
